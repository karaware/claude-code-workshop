# Claude Code セットアップ会 案

## このメモの位置づけ

`memo/20260709/info/memo_01.md` をもとに、2026/07/13 の週に実施する Claude Code セットアップ会の案を整理する。

4 名が同じタイミングで Claude Code を使い始める想定なので、単なるインストール会ではなく、最初から「安全に使える状態」を作る会にする。

## 会のテーマ

```text
Claude Code で仕事を加速させるための初期セットアップ
```

ただし、強調するのは以下。

```text
便利にする前に、先に制御する。
```

Claude Code は MCP を入れると一気に便利になるが、同時に外部サービスを操作できる範囲も広がる。

そのため、今回のセットアップ会では以下をセットで扱う。

- Claude Code を使える状態にする
- `settings.json` の考え方を理解する
- permissions / permission mode で危険操作を制御する
- 必要な MCP を導入する
- MCP を入れたあとに、何を許可し、何を確認し、何を拒否するか決める

## ゴール

### 最低ゴール

参加者全員が、以下の状態になる。

- Claude Code を起動できる
- 自分の作業リポジトリで Claude Code を使える
- `settings.json` がどこにあり、何を設定するものか説明できる
- permissions の `allow` / `ask` / `deny` の違いを説明できる
- MCP を 1 つ以上追加できる
- MCP は「便利な拡張」ではなく「外部サービスへの操作権限」だと理解する

### できれば到達したいゴール

- backlog MCP を使ってチケットを読める
- Slack MCP を使って情報取得できる
- aws-knowledge-mcp-server を使って AWS 公式知識を参照できる
- `.env` や `secrets` を読ませない deny 設定を入れる
- `terraform apply` / `destroy` などの危険操作を ask または deny にする
- auto mode を使う前に permissions を設計する、という順序を理解する

## 当日の流れ

### 1. 目的説明

最初に、セットアップ会の目的を明確にする。

```text
今日は Claude Code を入れるだけではなく、
仕事で安全に使い始めるための最低限の制御まで入れます。
```

伝えること。

- Claude Code は仕事を加速させる道具
- ただし、MCP を入れると外部サービスにも触れる
- 先に permissions / permission mode を理解しておく
- 危険操作を制御したうえで便利に使う

### 2. 起動確認

全員で以下を確認する。

- Claude Code が起動する
- 対象リポジトリを開ける
- 簡単な依頼ができる
- ツール実行時に確認が出ることを確認する

最初の依頼例。

```text
このリポジトリの構成を読んで、何のためのリポジトリか説明して
```

この時点では、変更作業ではなく read 系の依頼にする。

### 3. settings.json の説明

ここは長くしすぎない。

伝える中心は以下。

```text
settings.json は Claude Code に対する操作ルールを書く場所。
```

扱う項目。

- permissions
- permission mode
- hooks は概要だけ
- MCP の許可も permissions とセットで考える

細かい仕様説明より、実務での判断軸を優先する。

| 種類 | 判断 |
|---|---|
| allow | 調査、読み取り、差分確認など、自動実行してよいもの |
| ask | 適用、投稿、共有変更など、人間判断を挟みたいもの |
| deny | 機密情報、削除、本番変更など、絶対に止めたいもの |

### 4. MCP 導入

今回の本丸。

導入候補。

- backlog
- slack
- aws-knowledge-mcp-server

優先順位は以下がよい。

1. aws-knowledge-mcp-server
2. backlog
3. slack

理由。

- aws-knowledge-mcp-server は read / knowledge 系なので、初回導入に向いている
- backlog は業務情報を読む用途が分かりやすい
- slack は便利だが、投稿やチャンネル操作まで含む場合は制御を強く意識する必要がある

MCP の説明フレーズ。

```text
MCP は Claude Code に外部サービスを操作する手足を増やす仕組み。
入れたら終わりではなく、何をさせるかを permissions で決める。
```

### 5. 安全設定の追加

全員で最低限の設定を入れる。

#### 機密情報を読ませない

```json
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)",
      "Read(~/.aws/**)",
      "Read(~/.ssh/**)"
    ]
  }
}
```

#### Terraform の危険操作を確認制にする

```json
{
  "permissions": {
    "allow": [
      "Bash(terraform plan*)"
    ],
    "ask": [
      "Bash(terraform apply*)",
      "Bash(terraform destroy*)"
    ]
  }
}
```

#### MCP は read から始める

実際の tool 名は MCP サーバーごとに違うので、考え方として伝える。

```json
{
  "permissions": {
    "allow": [
      "mcp__*_search*",
      "mcp__*_read*",
      "mcp__*_get*",
      "mcp__*_list*"
    ],
    "ask": [
      "mcp__*_post*",
      "mcp__*_update*",
      "mcp__*_create*"
    ],
    "deny": [
      "mcp__*_delete*",
      "mcp__*_remove*"
    ]
  }
}
```

この設定例は、そのまま完成版として配るよりも、判断軸を説明するための例として扱う。

## 詰まりそうなところ

### MCP 導入で認証が詰まる

一番起きそう。

対策。

- 90 分枠にする
- 導入する MCP を 1 つに絞れるようにする
- 認証が詰まった人は、まず aws-knowledge-mcp-server のような read / knowledge 系に切り替える

### Slack MCP の権限が強すぎる

Slack は読み取りだけでなく投稿もできる可能性がある。

対策。

- 初回は read 系だけ許可する
- post 系は ask にする
- delete / update 系があれば deny にする

### settings.json の正確な場所で迷う

対策。

- 当日冒頭で全員の環境を確認する
- 個人設定とリポジトリ設定のどちらに置くかを説明する
- まずは個人の安全設定を作る方針にする

### auto mode をすぐ使いたくなる

対策。

以下の順序を強調する。

```text
1. default mode で使う
2. permissions を入れる
3. MCP を read 中心で入れる
4. 危険操作を ask / deny にする
5. 限定された作業から auto mode を使う
```
