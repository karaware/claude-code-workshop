# Claude Code セットアップ会の進め方

## 悩み

Claude Code のライセンスがセットアップ会の 1 日前に配布された場合、当日の開始時点で参加者ごとの状態がズレる可能性がある。

参加者は 4 名。

想定される状態。

- すでに Claude Code をインストールして触っている
- Claude Code は起動できるが、MCP は未設定
- ライセンスはあるが、まだ何も触っていない
- インストールやログインで詰まっている

さらに、求めているゴールも人によって違う可能性がある。

- まず Claude Code を起動したい
- Backlog MCP を使いたい
- permissions や安全設定を知りたい
- 実務でどう使うか相談したい

この状態で、全員を同じ手順で最初から最後まで進めると、進んでいる人は退屈になり、未着手の人は焦る。

## 結論

セットアップ会は、一斉ハンズオンではなく、チェックポイント方式にする。

```text
各自の現在地を確認し、
最低限安全に Claude Code + Backlog MCP を使える状態まで持っていく。
```

同じ作業を同じ速度で進める会ではなく、現在地別オンボーディングにする。

## 基本方針

当日冒頭で、参加者を状態別に分ける。

| グループ | 状態 | 当日の扱い |
|---|---|---|
| A | Claude Code 未インストール | インストールから進める |
| B | Claude Code は起動済み | MCP 設定から進める |
| C | MCP も触っている | permissions / 安全設定 / 実務ユースケースに進む |
| D | 詰まっている | 個別トラブルシュート対象 |

会の目的は、全員を同じ場所に完全に揃えることではない。

最低到達ラインを決めたうえで、進んでいる人には発展タスクを渡し、詰まっている人は詰まり箇所を特定する。

## 最低到達ライン

セットアップ会としての最低ラインは固定する。

```text
- Claude Code が起動できる
- Claude Code にログインできている
- /mcp を開ける
- Backlog MCP の設定方針が分かる
- Backlog の課題を read できる、または認証詰まり箇所が特定できている
- permissions で read / write / delete を分けて考える必要が分かる
```

「全員が Backlog MCP を完全に動かせる」ことを必達にしすぎない。

Backlog MCP は、API キー、スペース URL、権限、社内ネットワーク、MCP サーバーの仕様で詰まる可能性がある。

そのため、未完了でも以下なら成功扱いに含める。

```text
どこで詰まっていて、次に何を確認すればよいか分かっている。
```

## 早い人向けの発展タスク

先に進んでいる人を待たせないため、発展タスクを用意しておく。

- Backlog の自分の担当課題を Claude Code に要約させる
- 課題一覧から今日見るべきものを抽出させる
- permissions で Backlog の read 系を allow、update / comment 系を ask、delete 系を deny にする
- 自分の業務で使いたい Claude Code + Backlog の使い方を 1 つ書き出す
- Backlog 課題をもとに、作業手順案や調査観点を Claude Code に作らせる

早い人には、単に追加設定をさせるのではなく、実務ユースケースに寄せたタスクを渡す。

## 事前アンケート

前日または当日朝に、参加者へ現在地を聞く。

聞く項目。

```text
1. Claude Code のライセンスは使える状態ですか？
2. claude コマンドは実行できますか？
3. Claude Code にログイン済みですか？
4. Backlog API キーは用意できていますか？
5. 当日やりたいことはどれですか？
   - まず起動したい
   - Backlog MCP を使いたい
   - 安全設定を知りたい
   - 実務でどう使うか相談したい
```

この回答で、当日の支援順を決める。

## 当日の進行案

90 分の場合。

| 時間 | 内容 |
|---|---|
| 0:00 - 0:10 | 現在地チェック |
| 0:10 - 0:20 | 今日の最低ゴール説明 |
| 0:20 - 0:40 | 未セットアップ組: Claude Code インストール / 済み組: MCP 確認 |
| 0:40 - 1:00 | Backlog MCP 設定 |
| 1:00 - 1:15 | permissions と安全設定 |
| 1:15 - 1:25 | 各自の実務ユースケース確認 |
| 1:25 - 1:30 | 未完了事項と次アクション整理 |

## 当日の進め方

### 1. 現在地チェック

最初に全員へ確認する。

```text
claude --version が通る人？
claude にログイン済みの人？
/mcp を開いたことがある人？
Backlog API キーを持っている人？
```

この確認で、参加者を A / B / C / D に分ける。

### 2. 共通説明は短くする

全員に同じ説明を長くしない。

最初に伝えるのは以下だけでよい。

```text
今日は Claude Code をインストールするだけではなく、
Backlog MCP を安全に使い始めるところまでを目指します。
ただし開始時点の状態が違うので、チェックポイント方式で進めます。
```

### 3. チェックポイントごとに進める

チェックポイント。

```text
1. claude --version が通る
2. claude にログインできている
3. /mcp を開ける
4. backlog MCP が見える
5. Backlog の課題を read できる
6. permissions の方針を理解している
```

各自が今いるチェックポイントから進める。

### 4. write 系操作は当日やらない

初回セットアップ会では、Backlog の課題作成、コメント追加、ステータス更新、削除は扱わない。

扱うのは read 系だけにする。

理由。

- MCP の tool 名が環境によって違う
- 誤って業務データを書き換えるリスクがある
- 初回は permissions の考え方を理解するだけで十分

write 系は次回以降に、`ask` 設定を入れたうえで扱う。

## 主催側で用意するもの

- Claude Code インストール手順
- Backlog MCP 設定手順
- Backlog API キーの取得方法
- よくある詰まり一覧
- permissions のサンプル
- 早い人向け発展タスク
- 当日のチェックリスト

## 成功条件

セットアップ会の成功条件は、全員が同じ完成状態になることではない。

以下を満たせば成功。

- 未セットアップの人が Claude Code 起動まで到達する
- 起動済みの人が Backlog MCP の設定または確認まで進む
- 進んでいる人が permissions や実務ユースケースまで進む
- 詰まっている人は、詰まり箇所と次アクションが明確になる
- 全員が「MCP は外部サービス操作権限なので制御が必要」と理解する

## まとめ

セットアップ会は、一本道のハンズオンにしない。

4 人でも開始時点の状態はズレる。

そのため、以下の形にする。

```text
一斉ハンズオンではなく、現在地別オンボーディング。
最低到達ラインは固定する。
早い人には発展タスクを渡す。
詰まった人は、完了ではなく原因特定までを成功扱いにする。
```
